# 小提示

<p class="lead">能省时间的短笔记。没什么戏剧性，就是那些项目长出牙之后最容易忘的东西。</p>

<div class="example-strip">
<details class="example-card" open>
<summary>
<img src="guide-media/editor-add-menu.png" alt="Add menu screenshot">
<strong>给东西起诚实的名字</strong>
<p>如果节点或页面名字很含糊，以后的你就得花时间解码它。</p>
</summary>
<div class="example-body">
<p>起能说明角色的名字：root、toggle、preview、mask、page、icon、state。别的都只会让以后变考古。</p>
<div class="popup-note">烂名字不会让项目更酷，只会让编辑更慢。</div>
</div>
</details>

<details class="example-card" open>
<summary>
<img src="guide-media/value-link-fields.png" alt="Value link fields">
<strong>先检查链接，再叠更多逻辑</strong>
<p>坏掉的链接会很安静地躲在六个别的问题下面。</p>
</summary>
<div class="example-body">
<p>按钮没反应时，先看 value link、目标字段和控制台，再考虑重做整个菜单。</p>
<div class="popup-note">大多数 bug 不是魔法，通常只是少了一个赋值。</div>
</div>
</details>

<details class="example-card" open>
<summary>
<img src="guide-media/texture-slot-panel.png" alt="Texture slot screenshot">
<strong>把预览保持可见</strong>
<p>面板还在制作时，预览能告诉你到底接了什么。</p>
</summary>
<div class="example-body">
<p>如果你在接贴图，旁边最好留个预览。屏幕真的显示那个槽位时，更容易发现接错了。</p>
<div class="popup-note">看不见的设置最容易长出混乱。</div>
</div>
</details>

<details class="example-card" open>
<summary>
<img src="guide-media/shape-key-panel.png" alt="Shape key screenshot">
<strong>一次只测一个改动</strong>
<p>你改了五个东西，结果坏了一个，那你就造出了一部侦探小说。</p>
</summary>
<div class="example-body">
<p>改一个东西，导出一次，看看结果。它只是前期看起来慢，算上少掉的调试时间就不慢了。</p>
<div class="popup-note">系统稳定时，批量编辑当然行。活跃开发阶段，它只会把痛点藏起来。</div>
</div>
</details>
</div>
